# Production-readiness audit — 2026-05-10

A walk through the order → settle → close → shift → profitability flow plus
cross-cutting concerns. Findings are bucketed by severity. Items marked **fixed
in this pass** ship with this audit.

## Critical issues

### 1. ✅ Settle: overpayment leaves the order un-closeable — *fixed*

**Symptom**: User records cash + online payments that exceed total. Backend's
strict `balance == 0` check rejects close, but FE was happy with `balance <= 0`.
No way to undo a recorded payment, so the tab is stuck.

**Root causes**

- `RecordPayment` accepted any positive amount with no upper bound.
- `CloseOrder` requires exact zero (correct in finance terms, but FE didn't enforce it).
- No `DELETE` endpoint for payments, so mistakes were unrecoverable.

**Fix shipped**

- `RecordPayment` now rejects `amount > balance` with a clear error
  (`overpayment`) telling the user the remaining amount.
- New `DELETE /v1/orders/{id}/payments/{paymentId}` endpoint (open orders only,
  audit-logged).
- `SettleModal` shows an X button next to every recorded payment and a clear
  banner if balance < 0; close button disabled until balance is exactly zero.
- Reference-no input is now hidden for cash (no txn id needed); shown for
  online with placeholder "eSewa / Khalti id".

## High-severity issues (still open)

### 2. No idempotency on POST endpoints

`POST /payments`, `POST /orders/{id}/items`, `POST /shifts/open` etc. don't
accept an idempotency key. A flaky network + a double-click on **Add payment**
can create two `payments` rows. The new overpayment guard in §1 reduces blast
radius (the second one will 409), but a double-add of a partial payment within
the balance still slips through.

**Recommendation**: accept an `Idempotency-Key` header on POST payments and
order-items; store and short-circuit duplicates within a 24h window. Cheap to
add, prevents a class of payment-recording disputes.

### 3. PIN brute-force surface

`requireManagerOrApproval` checks the bcrypt-hashed PIN on every void/discount.
There's no lockout, throttle, or attempt counter. A 4-digit PIN has 10,000
combinations; a determined waiter can churn through it.

**Recommendation**: per-user attempt counter with an exponential cool-down (e.g.
3 fails → 30s lock, 5 fails → 5m, 7 fails → kick to admin reset). Or move to a
6-digit PIN with the same lockout policy.

### 4. No way to reverse a closed order

Once an order is closed, there's no refund / re-open / reversal flow. If a
customer disputes after close (common: "I was double-charged"), the manager has
no recovery path other than direct DB edit. Closing is irreversible by design,
but a refund concept is missing.

**Recommendation**: add a `refunds` table and a `POST /v1/orders/{id}/refund`
endpoint that records a negative payment + sets `order.refunded_at`. Don't
allow reversing inventory (already moved); just track the money.

## Medium-severity issues

### 5. Receipt printing — not implemented

There's no PDF render, no receipt printer integration, no email-receipt option.
Cafés in Nepal often print thermal receipts for VAT compliance.

**Recommendation**: backend endpoint that returns an HTML receipt + a print-CSS
stylesheet. Browser print works for 80mm thermal printers. Defer ESC/POS until
demand surfaces.

### 6. Reports use full table scans + recomputation

`/v1/reports/dashboard` and `/v1/reports/profitability` join `orders ×
order_items × menu_items` plus an aggregate of `expense_allocations` every
request. Indexed today — fine for one tenant with thousands of orders/month.
Past ~50k orders/month per tenant, latency will climb.

**Recommendation**: pre-aggregate daily KPI rollups into a `report_daily` table
on order close. Defer until measured slow.

### 7. Single-instance realtime hub

`realtime.Hub` is in-process. Two API replicas → events from replica A don't
reach clients connected to replica B. Single-replica deploy is fine; horizontal
scale isn't.

**Recommendation**: when scaling up, swap to Postgres `LISTEN/NOTIFY` (no new
infra) or Redis pub/sub. Don't pre-build it; the current design is correct for
single-replica.

### 8. Test coverage is thin outside multi-tenant isolation

The 7-test isolation suite is solid. There are no tests for:
- Payment math (tax base, rounding, half-cent edge cases)
- Order state machine (cancel-with-kitchen-items, void-after-close, etc.)
- Inventory link decrement on close
- Shift open/close flow + variance computation
- Profitability allocation math

**Recommendation**: at least one happy-path + one error-path test per
state-machine transition. Money math deserves its own table-driven test
(integer paisa rounding is easy to break subtly).

### 9. ✅ Shift screen UX — *fixed*

**Was**: bare `<label>`/`<input>` outside `.field` wrappers (poor spacing); no
warning if today's opening float ≠ yesterday's closing count; flat variance
display regardless of magnitude.

**Fix shipped**

- All form fields wrapped in `.field` for consistent typography & spacing.
- New `banner-info` showing previous shift's closing count when opening a new shift.
- Field-level warning (non-blocking) if entered float differs from previous close.
- Variance display now color-codes by severity (matched / minor ≤Rs 50 / warn
  ≤Rs 500 / bad >Rs 500) with contextual hints.
- Notes placeholder switches to "explain the variance — required for audit"
  when variance is large.

### 10. ✅ Profitability "100% margin" mystery — *fixed (UX) / open (auto-COGS)*

**Was**: a category with revenue but no expense allocations showed margin = 100%
with no explanation. Confusing because users assume the system tracks costs
automatically (it doesn't — allocation is manual via Expenses).

**Fix shipped (UX)**

- `ProfitabilityPage` now shows a banner explaining the calculation when any
  100%-margin category or unallocated COGS is present.
- 100%-margin cells render with an amber `AlertTriangle` and tooltip ("no
  COGS allocated") instead of a clean "100.0%".
- Banner deep-links into Expenses for one-click fix.
- New `docs/inventory-and-cogs.md` documents the relationship in detail.

**Still open**: implementing auto-COGS from inventory link
(`qty_consumed_per_sale × last_purchase_unit_cost_cents`) is deferred — see
inventory-and-cogs.md "Future work" for trade-offs.

### 11. ✅ Dropdown active-state visual artifact — *fixed (defensive)*

Without a screenshot, root cause is unconfirmed (likely a Chromium quirk where
inherited body styles paint a strikethrough on focused/selected `<option>`).
Added `text-decoration: none !important` guards on `select`, `option`, and
`.ssel-opt` (selected, hover, highlight states). If it persists, the next step
is to capture the rendered DOM in DevTools.

## Lower-severity issues

### 12. CORS allow-list is space-trim-only

`cfg.CORSOrigins` is read verbatim; no normalization. A trailing slash mismatch
(e.g. `https://example.com/` vs `https://example.com`) silently rejects
requests. Worth normalizing in `config.Load`.

### 13. No structured rate limiting on auth

`/auth/dev-login` has no rate limit. Dev-only, but if it ever leaks into prod
config, it's a brute-force surface. Worth a per-IP throttle on `/auth/*` with
`net/http`-friendly middleware (e.g. `golang.org/x/time/rate`).

### 14. Audit log is write-only — no UI

`audit_events` records denials, voids, discounts, payment removals, etc., but
there's no admin page to inspect it. Useful for post-incident review.

**Recommendation**: simple `/admin/audit` page (owner-only) with a filtered
list. Backend list endpoint with pagination.

### 15. Money-percentage parser is permissive

`parsePctHundredths` accepts `13`, `13.0`, `13.00` but silently returns 0 on
anything else (negative, scientific, multi-dot). The DB constraint is what
actually enforces the format, but the parser should validate up front and
return an error rather than silently zero-ing.

### 16. No backup automation

`docs/runbook.md` documents `pg_dump` but there's no scheduled job. Single
hardware failure = data loss. Out of scope for the app, in scope for the
deployment runbook.

### 17. Frontend bundle splitting

`vite build` bundles to one ~326 KB file. Code-splitting per route (Login,
PickWorkspace, AdminShell.*) would cut first-paint significantly on slow
mobile networks. Vite supports this via `lazy(() => import())`. Low effort,
moderate UX win on 3G.

### 18. Soft-delete inconsistency

Some tables use `deleted_at` (menu_items, menu_categories, inventory_items,
expenses) and others don't (payments, order_items, order_adjustments). The
absence is mostly correct (financial records shouldn't be soft-deleted) but
worth documenting that explicitly so a future migration doesn't add it
"for consistency".

## Cross-cutting checks (all good)

- ✅ Money math uses int64 paisa throughout. No float arithmetic on amounts.
- ✅ Tenant isolation via RLS, FORCE ROW LEVEL SECURITY, app_user role. 7-test
  isolation suite passes.
- ✅ Order/shift uniqueness enforced by partial unique indexes (one open tab
  per table; one open shift per tenant).
- ✅ All POST handlers run inside `TxMiddleware` so failures roll back atomically.
- ✅ WebSocket auth checks session cookie + tenant membership at upgrade time.
- ✅ Foreign keys cascade or restrict deliberately (payments → shifts is RESTRICT,
  preventing accidental shift deletion that would orphan cash).
- ✅ Inventory invariant: `qty_on_hand_units` is always derived from
  `stock_movements` via the trigger. Handlers cannot drift it.

## Suggested execution order if you fix all of these

1. **Idempotency keys** (§2) — protects payment integrity, low-risk add.
2. **Tests for payments + orders** (§8) — once you start changing things, you
   want a safety net.
3. **PIN lockout** (§3) — small, isolated, real security win.
4. **Receipt printing** (§5) — high user-visible value.
5. **Refunds / order reversal** (§4) — bigger lift, needs schema + UX.
6. **Auto-COGS toggle** (§10) — opt-in setting, doesn't disturb existing data.
7. **Audit log UI** (§14) — pair with §3 to see PIN failures in context.
8. The rest is "when scale demands it" — don't pre-build.
